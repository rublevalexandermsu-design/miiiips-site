# Agent Capability Catalog

## Как пользоваться этим каталогом

- Даешь одну бизнес-задачу, а не список skills.
- Я сам раскладываю ее по агентам, skills, страницам и данным.
- Этот каталог нужен, чтобы ты видел реальный диапазон задач и мог выбрать нужный уровень вмешательства.
- Формат ниже показывает: что агент умеет, к каким проектам относится, какие входы ему нужны, какие выходы он дает, и где его границы.

## 1. Global Orchestrator

### Проекты
- Институт глобализации
- АвтоСтатьи Кумской
- Курс ЭИ
- Гранты
- rowing_agent
- будущие ИИ-модули

### Skills
- `multi-agent-orchestrator`
- `institute-platform-architect`
- `cross-project-linker`

### Что умеет
1. Разбить большую задачу на подзадачи по проектам.
2. Назначить исполнителей и проверить, что зона ответственности не пересекается.
3. Решить, какие агенты нужны: один или несколько.
4. Собрать единый результат из разных модулей.
5. Установить порядок: архитектура -> контент -> реализация -> QA -> публикация.
6. Свести конфликтующие правки и выбрать источник правды.
7. Принять решение о выпуске только после QA.
8. Дать маршрутизацию по странице, домену или проекту.
9. Запустить параллельную работу нескольких агентов.
10. Зафиксировать, что было сделано и где лежит результат.

### Типовые задачи
- "Собрать новую лекцию по понедельникам и сразу довести до сайта."
- "Связать событие, публикацию и курс в одном выпуске."
- "Разложить новый research-case на sandbox, публикацию и грант."
- "Понять, какие проекты затронет новая карточка профиля."
- "Распределить задачи между Events, Editorial и QA."
- "Собрать план рефакторинга по всем страницам кабинетов."
- "Проверить, не сломает ли новая запись existing event flow."

### Выход
- План работ.
- Назначенные агенты.
- Согласованные изменения.
- Решение о выпуске.

### Границы
- Не должен писать массовый контент напрямую.
- Не должен брать на себя узкий доменный труд, если есть профильный агент.

## 2. Architecture Agent

### Проекты
- Главный сайт института
- Accounts shell
- Общая архитектура данных
- Cross-project reuse

### Skills
- `institute-platform-architect`
- `cross-project-linker`

### Что умеет
1. Определить, что становится общей платформенной сущностью, а что остается проектным модулем.
2. Спроектировать shell и связки между страницами.
3. Установить границы между public layer и internal layer.
4. Свести проекты в общий реестр данных.
5. Указать, где нужна общая навигация, а где отдельный маршрут.
6. Решить, какие страницы должны жить как шаблоны.
7. Нормализовать data contracts между проектами.
8. Сократить дублирование между разделами сайта.
9. Описать ownership для страниц и JSON-регистров.
10. Принять архитектурное решение перед внедрением.

### Типовые задачи
- "Собрать общую модель платформы из events, courses, grants и research."
- "Определить, что должно быть в shared shell."
- "Убрать дублирование между accounts и grants."
- "Сделать новую страницу как шаблон для будущих модулей."
- "Связать АвтоСтатьи и publications через общий data layer."
- "Разделить public-facing и internal-only сущности."
- "Определить, какой JSON должен быть источником правды."

### Выход
- Архитектурная схема.
- Список shared entities.
- Ownership map.
- Контракт страниц и данных.

### Границы
- Не исполняет массовый контент и не ведет сложные публикационные пайплайны.

## 3. Accounts Agent

### Проекты
- Институт глобализации
- Личный кабинет
- Курсы
- Гранты
- Публикации
- События

### Skills
- `account-shell-integrator`

### Что умеет
1. Вести профили участников, ролей и статусов.
2. Настраивать страницы кабинетов под разные роли.
3. Сводить маршруты пользователя: курс, событие, публикация, грант.
4. Показывать историю участия и progress state.
5. Разводить доступы и видимость блоков.
6. Создавать маршрут "войти -> увидеть доступ -> выполнить следующий шаг".
7. Поддерживать user-event history.
8. Связывать роль пользователя с дальнейшими агентами.
9. Работать с заявками и success-pages.
10. Держать shell без техтекста.

### Типовые задачи
- "Добавить новую роль в кабинет координатора."
- "Сделать участнику доступ к курсу и показать, что он уже завершил."
- "Показать в кабинете автора публикационный маршрут."
- "Связать профиль участника с историей событий."
- "Добавить карточку научного руководителя."
- "Показать, какие гранты доступны по профилю."
- "Сделать безопасный вход в институтский shell."

### Выход
- Обновленные кабинеты.
- Настроенные маршруты.
- Привязка роли к проектам.

### Границы
- Не должен писать редакционные материалы и research-case сам по себе.

## 9. Compliance Agent

### Проекты
- Внутренний контур отчётности АНО
- Telegram отчёты и напоминания
- Календарь сроков
- Архивы и доказательства подачи

### Skills
- `internal-compliance-orchestrator`

### Что умеет
1. Собрать актуальный список обязательных и плановых отчётов.
2. Разложить отчётность по месяцам и годовым архивам.
3. Отделить действующие обязательства от архивных периодов.
4. Поддерживать структуру папок по отчётам, месяцам и доказательствам.
5. Давать понятный Telegram-слой с кнопками, месяцами и статусами.
6. Напоминать о сроках подачи по рабочим дням и контрольным точкам.
7. Поддерживать ссылку между сайтом, ботом и архивом.
8. Не смешивать прошлые периоды с текущей обязательной отчётностью.
9. Отмечать документы как сданные, подтверждённые или требующие проверки.
10. Передавать выпуск на QA перед публикацией внутреннего контура.

### Типовые задачи
- "Обновить внутреннюю страницу отчётности по месяцам."
- "Добавить новый срок подачи и разослать напоминания."
- "Связать Telegram-вкладку с отчётным календарём."
- "Архивировать неактуальные годовые обязательства."
- "Проверить, что папки отчётов и доказательств совпадают с реестром."
- "Показать статус конкретного отчёта из внутреннего контура."
- "Собрать пакет для сдачи и фиксации подтверждения."

### Выход
- Обновлённый отчётный контур.
- Календарь по месяцам.
- Telegram mirror.
- Архив и папки доказательств.

### Границы
- Не должен менять правовой режим отчётности без подтверждённого источника.
- Не должен публиковать внутренний контур без QA.

## 10. QA Agent

### Проекты
- Весь public layer
- Все страницы и data registries перед выпуском

### Skills
- `site-content-qa-guard`

### Что умеет
1. Проверять публичный текст на техтекст.
2. Проверять карточки на пустые секции.
3. Проверять связи между страницами и данными.
4. Проверять CTA и маршруты.
5. Проверять account, course, event, grant, publication, research pages.
6. Проверять consistency между JSON и HTML.
7. Блокировать выпуск, если есть разрыв маршрута.
8. Проверять, что profile-aware блоки не пустые.
9. Давать читаемый отчет.
10. Не менять содержимое, а только разрешать или блокировать выпуск.

### Типовые задачи
- "Проверить, можно ли публиковать новый пакет страниц."
- "Найти техтекст в публикациях."
- "Проверить, не сломались ли ссылки в кабинете."
- "Сравнить JSON и HTML по событиям."
- "Снять выпуск только после успешной проверки."
- "Проверить новую research page перед пушем."
- "Проверить course page после добавления оплаты."

### Выход
- QA report.
- Pass / block decision.
- Список ошибок.

### Границы
- Не пишет контент и не проектирует архитектуру.

## 10. Как выглядят реальные пакеты задач

### Пакет для одного агента
- "Обновить кабинет автора."
- Владелец: Accounts Agent.
- Skills: `account-shell-integrator`.
- Выход: обновленный role page и маршрут автора.

### Пакет для нескольких агентов
- "Сделать лекцию по понедельникам, превратить ее в публикацию и связать с курсом."
- Агенты:
  - Events Agent
  - Editorial Agent
  - Courses Agent
  - QA Agent
- Выход:
  - event page
  - publication card
  - course link
  - release pass

### Пакет для всей платформы
- "Собрать новый research case, дать ему публичную страницу, связать с грантом и проверить выпуск."
- Агенты:
  - Architecture Agent
  - Research Agent
  - Grants Agent
  - Editorial Agent
  - QA Agent
- Выход:
  - public case page
  - grant route
  - публикационная связка
  - safe release
