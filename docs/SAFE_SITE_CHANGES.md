# Безопасные Правки Сайта МИИИИПС

Этот файл фиксирует рабочий порядок изменений сайта так, чтобы:
- не терять блоки, кнопки, шаблоны и маршруты;
- быстро находить, что исчезло;
- безопасно возвращать нужный кусок из Git-истории;
- не ломать рабочие формы, события, Timepad, Telegram, WhatsApp и JSON-подгрузки.

## 1. Базовый принцип

Все правки делаются в три шага:

1. Локальное изменение в рабочей копии.
2. Проверка:
   - `tools/site_release_guard.py`
   - `tools/site_data_guard.py`
   - визуальная проверка страницы
3. Коммит и push в GitHub.

Если изменение не закоммичено, Git не гарантирует его восстановление.

## 2. Что нельзя делать

- Не удалять блок только потому, что он мешает визуально, пока не проверено:
  - не использует ли его JS;
  - не тянет ли он данные из JSON;
  - не является ли он частью шаблона event/course/account page.
- Не заменять рабочие страницы полной пересборкой, если задача точечная.
- Не править сразу и HTML, и JS, и JSON без diff-проверки.
- Не публиковать без guard-проверок.

## 3. Что делать перед любой правкой

Перед изменением страницы или скрипта:

1. Посмотреть статус:
```powershell
git status --short
```

2. Посмотреть историю конкретного файла:
```powershell
git log --oneline -- "publications.html"
```

3. Посмотреть, что менялось внутри файла:
```powershell
git diff -- "publications.html"
```

4. Если правка затрагивает публичный текст, отдельно проверить:
- нет ли технических слов;
- не пропала ли кнопка;
- не исчез ли mount-блок, если он реально нужен.

## 4. Обязательные проверки перед публикацией

Из корня сайта:

```powershell
python .\tools\site_release_guard.py
python .\tools\site_data_guard.py
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\publish_with_guard.ps1
```

Публикация допустима только если все проверки дают `ok` или `release gates passed`.

## 5. Если что-то случайно исчезло

Типовые случаи:

### Исчезла кнопка
- сначала проверить, не был ли отключён JS, который её создаёт;
- потом посмотреть последний рабочий коммит по файлу;
- затем восстановить нужную версию файла или вынуть оттуда нужный фрагмент.

### Исчез блок на странице
- проверить HTML на удалённый контейнер;
- проверить JS на отключённый mount;
- сравнить с последним рабочим коммитом.

### Страница визуально развалилась
- проверить, не была ли удалена локальная CSS-поддержка;
- проверить, не исчезли ли классы, обёртки или внешние стили;
- при необходимости временно вернуть прошлую рабочую вёрстку из Git и затем дочистить её аккуратно.

## 6. Как восстанавливать безопасно

Использовать не прямой откат всей ветки, а точечное восстановление:

### Вариант A. Посмотреть старую версию файла без перезаписи
```powershell
git show <commit>:./publications.html
```

### Вариант B. Выгрузить старую версию файла в preview
```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\restore_site_asset.ps1 -Commit <commit> -RelativePath publications.html -Mode preview
```

### Вариант C. Явно восстановить файл из конкретного коммита
```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\restore_site_asset.ps1 -Commit <commit> -RelativePath publications.html -Mode restore
```

Важно:
- сначала использовать `preview`;
- `restore` делать только когда понятно, что нужна именно та версия.

## 7. Как восстанавливать не весь файл, а только блок

Порядок:

1. Найти коммит, где блок ещё был.
2. Выгрузить файл в preview.
3. Сравнить текущий файл и preview.
4. Вернуть только нужный HTML/JS-фрагмент вручную, без отката остальных правок.

Это предпочтительнее, чем откатывать всю страницу.

## 8. Что считать “стабильным шаблоном”

Стабильными считаются:
- event pages;
- course pages;
- account pages;
- формы;
- карточки и листинги событий;
- кнопки регистрации и Timepad;
- блоки контактов;
- шаблон понедельничных лекций.

Если меняется один из этих шаблонов, после правки нужно:
- проверить сам шаблон;
- проверить минимум одну страницу, использующую его.

## 9. Как действовать при большом пакете изменений

Если правок много:

1. Сначала менять только тексты.
2. Потом отдельно менять JS-вставки.
3. Потом отдельно менять JSON-подписи.
4. После каждого слоя прогонять guards.

Так проще понять, где именно что сломалось.

## 10. Что делать, если замечена потеря блока уже после публикации

Порядок реакции:

1. Найти страницу и точный пропавший элемент.
2. Проверить последний коммит, который его затрагивал.
3. Сравнить рабочую версию с предыдущей.
4. Вернуть только нужный фрагмент или включение нужной логики.
5. Снова прогнать guards.
6. Опубликовать точечный hotfix отдельным коммитом.

## 11. Главный принцип восстановления

Если что-то потерялось, это не повод пересобирать страницу заново.

Правильный путь:
- найти, где это уже работало;
- вернуть это из Git;
- адаптировать под текущие правки;
- не терять остальной рабочий слой.
